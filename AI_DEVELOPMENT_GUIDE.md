## 1. 心構え：コーダーから「コマンダー（司令官）」へ

AI時代の開発では、あなたがゼロからコードを書く必要はありません。あなたの役割は以下の3つに変化します。

1.  **指示する (Prompt)**: 「こういう機能を作って」と具体的に伝える。
2.  **確認する (Review)**: AIが出してきたコードが、やりたいことと合っているか確認する。
3.  **責任を持つ (Judge)**: AIの提案を採用するか、修正させるか、却下するかを決める。

> **ポイント**: AIは優秀ですが、たまに平気で嘘をついたり、危険なコード（セキュリティ無視など）を書いたりします。「新人の優秀なプログラマー」だと思って、必ずダブルチェックする姿勢が重要です。

## 2. 実践フロー：各工程での「AIへの指示」と「人間の判断」

### ① 実装（ドラフト作成）

**目的**: 白紙の状態から、動くたたき台（ドラフト）を最短で作る。

- **AIへの指示**:
  - 「〇〇画面の入力チェック処理を書いて」
  - 「CSVファイルを読み込んでリストにするクラスを作って」
- **人間の判断ポイント（ここを見る！）**:
  - **日本語の違和感**: 変数名やコメントがおかしくないか？（例: `var kazu = ...` など変な名前になっていないか）
  - **仕様との不一致**: 「必須項目」と言ったのにチェックが抜けていないか？
- **PMのアクション**:
  - 細かい文法がわからなくても、**「やりたい処理の流れ（ロジック）」**が合っているかを確認してください。

### ② テスト・修正（トラブルシューティング）

**目的**: エラーを解消し、正しく動く状態にする。**ここが最大の山場です。**

- **AIへの指示**:
  - エラーログをそのまま貼り付けて、「原因と修正案を教えて」と聞く。
- **人間の判断ポイント（ここを見る！）**:
  - **「臭い物に蓋」をしていないか？**: AIは時々、「エラーが出るテスト行を削除しましょう」や「チェックを無効化しましょう」という提案をします。これは**却下**してください。
  - **副作用の有無**: 修正することで、他の機能が壊れそうではないか？（例：全体の文字コード設定を変えるなど、影響範囲が広そうな提案は要注意）
- **PMのアクション**:
  - AIは修正案を複数出してくることがあります（案A、案B...）。
  - それぞれの「メリット・デメリット」を聞き出し、**「今のプロジェクトの方針に合うのはどっちか？」** をあなたが決断します。

### ③ レビュー（品質チェック）

**目的**: 誰が見ても分かりやすく、安全なコードにする。

- **AIへの指示**:
  - 「このコードの複雑な部分を解説して」
  - 「可読性を上げるためにリファクタリング（整理）して」
- **人間の判断ポイント（ここを見る！）**:
  - **機密情報**: パスワードや顧客名、社内サーバーのIPアドレスなどがコードに直接書かれていないか？
  - **属人化**: AIが書いた高度すぎるコードは、後で誰も直せなくなります。「もっと平易な書き方にして」と突き返す勇気を持ってください。

## 3. 職場での活用法：GitHub Copilot Chat 操作マニュアル

職場のVS CodeやVisual Studioには、ブラウザ版のAI（GeminiやChatGPT）がいません。
その代わり、エディタに内蔵された **「GitHub Copilot Chat」** を使います。

### 📌 重要なコツ：AIに「文脈（コンテキスト）」を読ませる方法

ブラウザ版と違い、ファイルをアップロードする必要はありません。代わりに以下の手順を踏みます。

1.  **関係するファイルを開く（タブに表示する）**
    - Copilotは、**「現在エディタで開いているタブ」** を見て回答を生成します。
    - 例：エラー修正なら、「エラーが出ているコード」と「テストコード」の両方を開いておきます。
2.  **範囲選択（ハイライト）する**
    - 特に見てほしい行をマウスでドラッグして選択してからチャットを開くと、精度が劇的に上がります。
3.  **チャットを開く**
    - VS Code: 左側のチャットアイコン、または `Ctrl + I`（インラインチャット）
    - Visual Studio: `Alt + /` など

### 📌 修正適用の流れ

1.  チャットで質問する（「このエラーを直して」）。
2.  Copilotがコードを提案してくる。
3.  **【最重要】差分（Diff）を確認する**:
    - いきなり「適用」を押さず、元のコードとどう変わるのかを表示させます。
    - 「削除される行（赤）」と「追加される行（緑）」を見比べます。
4.  納得したら「適用（Insert/Accept）」ボタンを押す。

## 4. 【付録】コピペで使える実務用プロンプト集

技術的な用語が思い浮かばなくても大丈夫です。以下のテンプレートの `[ ]` 部分を埋めて、Copilot Chatに投げてください。

### A. 何から始めればいいか分からない時（実装）

> **使い所**: 新しい機能を作り始める時、たたき台を作らせる。

```text
# 役割
あなたはC#開発の専門家です。

# 依頼
以下の要件を満たす [クラス名/機能名] のコードを作成してください。

# 要件
- 目的: [何をする機能か。例：CSVファイルを読み込んで画面に表示する]
- 入力データ: [例：パス文字列]
- 出力データ: [例：Productクラスのリスト]
- ルール:
  - エラーハンドリング（Try-Catch）を入れること
  - 変数名は分かりやすい英語にすること
  - 各メソッドには日本語のコメントを入れること
```

### B. エラーが出て困った時（修正）

> **使い所**: ビルドエラーやテスト失敗時。**エラーログのコピペが最強です。**

```text
# 現象
コードを実行したところ、以下のエラーが発生しました。

# エラーログ
[ここにエラーメッセージを貼り付ける]

# 依頼
1. このエラーが発生している原因を、初心者にもわかるように解説してください。
2. その上で、修正後のコードを提示してください。
3. もし修正案が複数ある場合は、それぞれのメリット・デメリットも教えてください。
```

### C. コードの意味が分からない時（解説）

> **使い所**: 難しそうなコードを解読する時や、引き継ぎ資料を作る時。
> **手順**: 分からないコードをマウスで選択してから実行。

```text
# 依頼
選択したコードが何をしているのか、行ごとに日本語で解説してください。
また、もしこのコードにバグの可能性や、セキュリティ上の懸念があれば指摘してください。
```

### D. テストコードを書きたい時（テスト）

> **使い所**: 実装はできたが、テストコードを書くのが面倒な時。
> **手順**: テスト対象のファイルを開いた状態で実行。

```text
# 依頼
現在開いている [ファイル名] の機能をテストするための、単体テストコード（xUnit）を作成してください。

# テストケースの要件
1. 正常系: 正しいデータを入れたら成功すること
2. 異常系: 空文字やnullを入れたらエラーになること
3. 境界値: 最大文字数（例: 256文字）のデータでも動作すること
```
